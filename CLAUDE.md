# Biocontainers Project - Claude Reference

## Project Overview

Purdue RCAC Biocontainers is a collection of containerized bioinformatic applications deployed as Lmod modules on Purdue HPC clusters (Bell, Gautschi, Gilbreth, Negishi, Scholar) and ACCESS Anvil. Each app is a Singularity/Apptainer container wrapped by a Lua modulefile. Documentation is published at https://biocontainer-doc.readthedocs.io.

## Repository Structure

```
modulefiles/          # Lmod modulefiles, one .lua per app version per cluster
  <cluster>/          # anvil | bell | gautschi | gilbreth | negishi | scholar
    <appname>/
      <version>.lua   # e.g. samtools/1.17.lua
      default         # symlink to the default version .lua

docs/                 # Sphinx RST documentation
  index.rst           # auto-managed TOC (Application list section regenerated by scripts)
  source/
    <appname>/
      <appname>.rst   # one RST page per app

definition_files/     # Singularity .def files for custom-built containers (not from BioContainers)

scripts/
  documentation/      # doc automation (run `make` from here)
  pull_and_module/    # helpers to pull images and generate modulefile stubs
  ood_scripts/        # Open OnDemand app templates
```

## Modulefile Format (`modulefiles/<cluster>/<app>/<ver>.lua`)

- MIT-licensed Lua files for Lmod
- Key variables every modulefile must define:
  - `image` — SIF filename (e.g. `staphb_samtools:1.17.sif`)
  - `uri`   — Docker URI (e.g. `docker://staphb/samtools:1.17`)
  - `programs` — Lua table of executable names the module exposes
  - `entrypoint_args` — extra args inserted before the program name (often `""`)
- Each program gets a shell function via `set_shell_function` that runs `singularity run <image> <program> "$@"`
- GPU support: auto-detects Nvidia (`--nv`) and AMD (`--rocm`) at load time
- `conflict(myModuleName())` prevents multiple versions loaded simultaneously
- Respects env vars:
  - `BIOC_IMAGE_DIR` — directory containing pre-pulled SIF files
  - `BIOC_SINGULARITY_MODULE` — name of Singularity/Apptainer Lmod module to load (set to `"none"` to skip)

### SIF image naming convention
Replace `/` with `_` in the registry path and append `.sif`:
- `docker://staphb/samtools:1.17` → `staphb_samtools:1.17.sif`
- `docker://quay.io/biocontainers/vcftools:0.1.16--h9a82719_5` → `quay.io_biocontainers_vcftools:0.1.16--h9a82719_5.sif`

### Modulefile version naming convention
Strip `--<hash>_<release>` suffixes from biocontainers URIs; keep interpreter tags (py, pl):
- `vcftools:0.1.16--h9a82719_5` → modulefile `0.1.16.lua`
- `bowtie2:2.4.2--py36hff7a194_2` → modulefile `2.4.2-py36.lua`

## Adding a New App or Version

1. **Pull the image and generate a modulefile stub** (from `scripts/pull_and_module/`):
   ```bash
   ./bioc_pull_and_module.sh docker://<registry>/<app>:<version>
   # or separately:
   ./bioc_pull2sif.sh docker://<uri>          # pulls .sif
   ./bioc_pull2mod.sh docker://<uri>          # generates stub .lua
   ```
2. **Manually edit the stub** — fill in: app name, description, homepage URL, `programs` list, `entrypoint_args`, conflicts, any custom env vars.
3. **Place the .lua file** under `modulefiles/<cluster>/<appname>/<version>.lua` for each cluster it should be available on.
4. **Update documentation** (see below).

## Documentation Update Workflow

All three scripts must be run from `scripts/documentation/`. Running `make` executes them in order:

```bash
cd scripts/documentation
make
```

### Step-by-step (what `make` does):

1. **`update_apps_inventory.sh -v`**
   - Scans all `modulefiles/` for `.lua` files and `default` symlinks
   - Outputs `rcac_biocontainers_inventory.json` — maps each app to its per-cluster versions
   - Skips: `*.modulerc.lua`, hash-suffixed files, `modtree/`, `rocmcontainers/`, `ngc/`, `Core/`

2. **`generatelistofmissingfiles.sh`**
   - Compares `modulefiles/<cluster>/` dirs against `docs/source/`
   - Writes `listofmissingfiles.txt` — apps that have modulefiles but no RST doc yet

3. **`generatedocumentation.sh`**
   - For each app in `listofmissingfiles.txt`, creates `docs/source/<app>/<app>.rst`
   - Parses the latest `.lua` to extract description, links, and programs list
   - Also regenerates the `Application list` section of `docs/index.rst`

4. **`appendversions.sh`**
   - Updates the `Versions` table in every `docs/source/<app>/<app>.rst`
   - Reads per-cluster version data from `rcac_biocontainers_inventory.json`

### RST doc structure (per app):
```
Introduction (from lua help block)
| For more information links
.. note:: citation reminder
Versions (list-table by cluster, auto-updated)
Commands (from programs{} in lua)
Module (load instructions)
Example job (.. tab-set:: with tabs: Anvil, Bell/Gautschi/Negishi, Gautschi-AI, Gilbreth, Scholar)
```

## Notes & Gotchas

- Scripts in `scripts/documentation/` use `cd ../../` to resolve the repo root — they must be run from that directory (or via `make`).
- `generatedocumentation.sh` uses the **alphabetically last** `.lua` file in a folder as the source for description/commands, so the highest version is typically used.
- `appendversions.sh` replaces the content between `Versions` and `Commands` headings — do not change those heading names in RST files.
- Modulefile stubs generated by `bioc_pull2mod.sh` use `singularity exec` (not `run`); production modules may use `singularity run` — verify after generation.
- The `docs/index.rst` Application list section is fully regenerated on each `make` run; do not manually edit below the `Application list` caption.
- Custom-built containers (when upstream images are broken) have their `.def` files in `definition_files/`.
- Shebang `#!/bin/sh -l` breaks some biocontainer modules; example jobs always use `#!/bin/bash`.
